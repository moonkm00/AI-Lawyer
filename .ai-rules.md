## 1. Domain-Driven Package Structure (도메인 주도 패키지 분리 규칙)

**DO NOT modify or create files in other domains unless explicitly approved by the human developer.**

## 2. Layered Architecture (계층형 아키텍처 원칙)
Inside each domain package, you must strictly adhere to the 4-tier layered architecture. **Do NOT mix concerns.**

- `controller`: Only handles HTTP requests, DTO mapping, and returning HTTP responses. **NO Business Logic here.**
- `service`: Contains all core business logic, algorithmic operations, and exception throwing.
- `repository`: Exclusively for Spring Data JPA Repository interfaces interacting with the Database (Supabase).
- `domain` (or `entity`): JPA Entities representing Database tables, and Request/Response DTO classes.

## 3. Collaboration & Communication (협업 충돌 방지 및 PR 가이드)
- **Shared Entities**: When modifying core, shared entities (like `Member`, `User`), you must explicitly confirm with the human developer that this change was discussed and agreed upon with the rest of the team. Do not add random columns without permission.
- **Pull Requests (PR)**: When generating commit messages or PR descriptions, clearly state the domain scope you touched (e.g., `feat(ai): integrate LangChain4j for PDF parsing`).
- **Database (Supabase)**: We are using Supabase (PostgreSQL). Use standard SQL and JPA conventions. `spring.jpa.hibernate.ddl-auto=update` is used for development, meaning entity changes automatically sync to the database.

## 4. Frontend Interaction Rule
- If the frontend (Next.js) needs an API, the backend Developer MUST define the Response DTO shape clearly within their domain package and only expose that REST endpoint through the `controller`.
